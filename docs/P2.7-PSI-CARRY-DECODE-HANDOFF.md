## 1. The problem (one paragraph)

Under HF's cached `generate()`, the seam sees only the newest token's hidden state each step, so the
latch's `cummax` runs over length 1 and ψ **resets every token** — `qnm_block.py` calls `init_state`
(→ zeros) on every forward and discards the advanced register (`delta_world, _`). The constraint never
accumulates: a violation in the prompt or three tokens back has zero effect on the enforce delta now,
defeating the entire point of ψ₀ being trajectory-dependent. Correct ψ currently exists **only** on Path B
(`greedy_generate`), which re-forwards the whole prefix every step (O(T²)) — too slow for the ~27K-token
long-context probes at 8B. P2.7 makes ψ both correct **and** fast. Full detail:
`DECODE-AND-PSI-PERSISTENCE.md` §3.

---

## 2. Decision (pinned — do not relitigate)

Use a **custom batch-1 greedy decode loop** (design-doc option §8.1). **Not** HF `generate()` with the seam
flipped on (runs, produces tokens, silently wrong — §3). **Not** riding HF's `Cache` object (option §8.3 —
tightest coupling to transformers internals, which churn across versions; unacceptable under a load-bearing
path).

Rationale to carry forward:

- The seam is a layer buried **inside** `model.forward` — it is the swapped `blocks[L]`. So ψ **cannot** be
  passed as a forward kwarg: HF calls each decoder layer with a fixed signature and won't thread an unknown
  arg down to layer L. The carry therefore **must** live as state on the seam.
- The only real choice is *who controls it*: infer prefill-vs-decode from `seq_len == 1` (fragile — a
  1-token prompt misreads) **or** have the loop set it explicitly. **The loop sets it explicitly.** That is
  option 8.1, and it is option 8.2 minus the footgun.
- A custom loop touches only the public `forward(use_cache=True, past_key_values=...)` contract, never
  `Cache` internals.

## 3. Implementation

Three changes. Names provisional — reconcile per §0.

### 3a. Seam: seed ψ from a carry instead of zeroing every call

`QNMBlock.forward`, enabled branch. Replace the §3 bug (`init_state` every call + discard `psi_next`):

```python
# __init__: add the carry slots
self.psi_in = None      # set by the decode loop; None ⇒ fresh sequence (prefill start)
self.psi_out = None     # advanced register, read by the loop after each forward

# forward (enabled branch):
c = self.fabric.surface(h_base)                                    # (B, T)
psi = (self.world_state.init_state(h_base.size(0), h_base.device)  # psi_in None ⇒ zeros (fresh)
       if self.psi_in is None else self.psi_in)                    # else carried seed
delta_world, psi_next = self.world_state(h_base, psi, c)           # keep psi_next (was discarded)
self.psi_out = psi_next                                            # expose for the loop
delta_fabric = self.fabric(h_base)
h_out = h_base + self.gate_fabric * delta_fabric + self.gate_world * delta_world
return h_out
```

**Invariant to preserve:** when `psi_in is None`, the forward **output must be identical to today's code**
(fresh `init_state` zeros over the full sequence). The only change on that path is keeping `psi_next`
instead of discarding it, which does not affect `h_out` or `delta_world`. Therefore every full-sequence
forward — training, the replica test, Path B — is unchanged. Only the decode loop ever sets `psi_in`
non-None. Confirm nothing else mutates these slots. (`psi_next` is already `.detach()`-ed per
`candidate.py:63`, so storing it introduces no autograd leak.)

### 3b. New decoder: `greedy_generate_psi`

Add alongside `greedy_generate` in `eval/arms.py` (or a small `eval/decode_psi.py` — implementer's call).
Batch-1 greedy, carries the latch with the cache:

```python
@torch.no_grad()
def greedy_generate_psi(core, seam, input_ids, max_new_tokens, eos_id=None):
    seam.psi_in = None                              # fresh sequence ⇒ prefill seeds ψ at 0
    out  = core(input_ids, use_cache=True)          # seam runs over all T prompt positions;
    past = out.past_key_values                      #   steers prompt h ⇒ steered K/V cached (§3c)
    m    = seam.psi_out                             # m_T captured over the prompt trajectory
    nxt  = out.logits[:, -1].argmax(-1, keepdim=True)
    ids  = [nxt]
    for _ in range(max_new_tokens - 1):
        if eos_id is not None and (nxt == eos_id).all():
            break
        seam.psi_in = m                             # seed latch with carried running-max
        out  = core(nxt, past_key_values=past, use_cache=True)
        past, m = out.past_key_values, seam.psi_out # advance + carry
        nxt  = out.logits[:, -1].argmax(-1, keepdim=True)
        ids.append(nxt)
    return torch.cat(ids, dim=1)
```

The exact `core(...)` call — return type, how `past_key_values` is passed back, whether `cache_position` is
required — is the **version-dependent** part. Match the pinned `transformers`. Greedy `do_sample=False`
decoding must be identical to Arm 0/P (same instrument).

### 3c. Prefill steering is required, not optional

`psi_in = None` at prefill means the seam runs over all T prompt positions and steers the prompt's own
hidden states → the cached K/V are the **steered** ones. This is mandatory: Path B (the oracle) re-forwards
and re-steers the prompt every step, so to match it, the cached path must steer at prefill and cache the
steered K/V. "Steer only generated tokens" would **not** equal the reference and silently breaks the gate.
Steering is output-side at layer L (`h_out = h_base + …`), so it perturbs K/V for layers **≥ L only** —
block L's own K/V come from its pre-steer input. Confirm there is a **single injection point** and no second
steer below L.

### 3d. Wire into the Arm-A path

In the arm-runner dispatch (`eval/arms.py` ~115–161), the `arm == "A"` branch must call
`greedy_generate_psi` instead of HF `generate()`. Confirm `--tau` and the trained ReZero `gate_world` are
loaded **from the checkpoint** (`load_arm_a_model`), not defaulted — both are part of the carried
computation.

## 4. Correctness gate (non-negotiable — the KV analogue of `test_bit_identity`)

New test: **the cached ψ-decode must produce token-identical output to `greedy_generate` (Path B), seam
on.**

- Compare **token IDs (argmax)**, not tensor `allclose`. Float-associativity noise must not fail a correct
  cache, but any real divergence — forgotten latch, missed reset, mis-seeded step — flips a token and the
  test catches it.
- **Run target: the tiny-random HF core already used in CI, not `TinyTransformer`.** Reason:
  `greedy_generate_psi` exercises `past_key_values`, which `TinyTransformer` likely does not implement
  (Path B exists *because* it re-forwards instead of caching). The tiny-random HF model exercises the
  **real** cache mechanics on a model small enough that Path B is cheap.
  **Reconcile:** confirm `greedy_generate` (or an equivalent full-prefix forward) runs on the HF core; if it
  is `TinyTransformer`-only today, generalize it so the oracle exists where the test needs it.
- Run on **CPU float32** for determinism (consistent with the existing bit-identity discipline).

Until this test is green, Arm A may run on Path B (slow, ψ-correct by construction) — keep that fallback
available so Arm A is not blocked if the cache work runs long.

**Do not weaken existing discipline:** `tests/test_bit_identity.py` stays `torch.equal`; the default
World-State stays `NoOpWorldState` (literal zeros); the §3a change must not alter full-sequence behavior.
